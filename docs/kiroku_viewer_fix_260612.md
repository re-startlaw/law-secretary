## 課題③ 検索結果クリックで文書に飛ばない【最優先】

### 原因（コード確認済み）
1. `app.js` の `window.__openDoc`（app.js:981-991）が `state.evidenceTab` のみリセットし、`state.filter` / `state.annoOnly` を残すため、絞り込み入力が残っていると `visibleDocuments()`（app.js:98-109）で該当行が描画されない。
2. 展開後に該当行へ `scrollIntoView` していない（タブは切り替わるが画面上変化が見えない）。
3. ファイル改名後は索引行（旧 rel_path）と未索引行（新名）が同一 sha256 で二重に並び、`renderTable` が同一 sha の全行にビューアを二重マウントする（app.js:210-240）。

### 修正
- `__openDoc(sha, page)`:
  - `state.filter = ""`、`state.annoOnly = false` にリセットし、**DOM側も同期**（`#filter-input.value = ""`、`#anno-only.checked = false`）。
  - `renderTable()` 後に `tr.doc-row[data-sha=...]` へ `scrollIntoView({block:"start"})`。
  - `state.documents` に sha が無い場合は黙って return せず、`loadDocuments()` を1回挟んで再試行 → それでも無ければ「文書一覧に見つかりません。再索引・改名の可能性があります」と表示。
- `server.py build_listing()`: 同一 sha256 の重複行を排除。**ただし索引行の実ファイル（base/rel_path）が存在しない場合のみ**未索引行を採用する。同一内容のPDFが別名・別フォルダに正規に2部存在するケース（正当）は両方残す。
- 受け入れ条件: 検索ヒットをクリック → 文書DBタブに切替 → 該当行が画面内に展開 → ビューアが**ヒットしたページ**で開く、まで。あわせて TXT 表示パネルでヒット語をハイライト（軽実装・スキャンPDFの当たり確認用）。

### テスト
- `build_listing` の重複排除: 「索引行ファイル欠落＋同 sha の walk 行」→1行になる／両ファイル実在→2行残る。

## 課題② 弁護革命ファイル名末尾の英数字ID除去

### 原因（コード確認済み）
- 弁護革命はファイル名末尾に `.xxxxxxxxxxxx`（12桁・小文字英数字混在のID。例 `d73ca7qjaj7b`）を付ける。`Path.stem` は最後の拡張子しか落とさないため title に残る。
- `parse_name_only`（evidence_index.py:182-210）の作成者分離が全角＠のみ対応（197行目）。弁護革命は半角 `@`。
- ID中の6連続数字が `DATE_RE` の compact パターン（evidence_index.py:65）に誤マッチし、**document_date が汚染されている可能性**。

### 修正
- `scripts/evidence_index.py parse_name_only` 冒頭で末尾IDを除去:
  - パターン: `\.(?=[a-z0-9]*[0-9])(?=[a-z0-9]*[a-z])[a-z0-9]{10,16}$`（英字・数字の混在必須、10〜16桁で長さ違いにも耐える）。
  - 除去後の残りが空になる場合は除去しない。
  - 処理順: **ID除去 → 符号 → @分離 → 日付抽出**（誤日付汚染の防止）。
- 作成者分離を `[＠@]` の**最後の出現**で rsplit（全角・半角両対応）。
- `server.py build_listing()`: 索引済み行はファイル名にIDパターンを検出した場合、title の正規表現除去ではなく **`ei.parse_name_only(Path(file_name).stem)` で4項目（符号・標目・日付・作成者）とも再計算して上書き**（DB内の誤日付・誤作成者も表示時に直る。再索引不要・D3整合）。
- `api_search` のヒット（`d.title` 由来）にも同サニタイズを適用。
- 原本ファイル名は**改名しない**（弁護革命側の文書キーのため）。README の D13 節に「弁護革命由来のファイル名は改名しない（表示上だけIDを隠す）」を明文化。
- 【確認事項1】`@署名要確認` のようなステータス語が作成者列に入る問題: ステータス語リスト（署名要確認・要確認・未確認 等）に当たる場合は作成者列でなくメモ相当の表示に振り分けるか、米谷弁護士に確認の上決定。

### テスト
- 除去すべき例（観測7例: `d73ca7qjaj7b` `6ggnjsv9fvd1` `20pegmr6ib64` 等）／除去してはいけない例（`報告書.最終版`、`xxx.20240101`（純数字日付）、日本語標目のみ）の両方をpytestに。
- 半角@・全角＠の作成者分離、ID除去後に日付誤抽出が起きないこと。

## 課題① ウィンドウ幅へ追従しない

### 原因（コード確認済み）
- `viewer.js fitWidth()`（453-460行）は load 時のみ。リサイズ監視なし。
- CSS: flex の `min-width:auto` により `.viewer-right`（style.css:165）と `.pdf-scroll` が canvas 固定px幅以下に縮まず、テーブルを押し広げる。
- **既存構造の問題（エンジニアレビュー指摘・先行修正必須）**: `destroy()` を呼ぶのは `navDoc()` のみ。`renderTable()` は `tbody.innerHTML=""` でDOMだけ消し、開いている文書に毎回新 ViewerInstance を作り直すため、旧インスタンス（loadingTask・タイマー）がリークする。ResizeObserver を足す前にここを直さないと disconnect が永遠に呼ばれない。

### 修正
1. **ライフサイクル整理（先行）**: `app.js` にマウント中インスタンスの `Map<sha, viewerInstance>` を保持。`renderTable()` 冒頭・`closeDoc()` で旧インスタンスの `destroy()` を呼んでから再構築。`mountInlineViewer` がインスタンスを返すよう変更。
2. `viewer.js` に `fitMode` フラグ（初期 true。±/100% で false、「幅」で true）。
3. `ResizeObserver` を `scrollEl` に装着。fitMode 時のみ debounce 200ms で `fitWidth()` → `reflow()`。
   - **無限ループ対策**: 前回フィット時の幅を記録し、差が2px以下なら何もしない（スクロールバー出現/消失による再発火対策）。
   - reflow 後に `jumpTo(this.currentPage)` で表示ページを維持。
   - `destroy()` で `disconnect()`。
4. CSS: `.viewer-right { min-width: 0; }`、必要に応じ `#docs-table { table-layout: fixed; }`（列幅指定の再調整込み）。
- 手動ズーム中はリサイズ非追従（注釈作業中に倍率が飛ばないこと）。

### 検証（pytest不可・Claude Previewで目視）
- ウィンドウリサイズ → canvas 追従／±ズーム後はリサイズ非追従／「幅」ボタンで追従復帰／リサイズ後も表示ページ維持。

## 課題⑤ 事件フォルダの新規作成

### 修正
- `server.py`: 専用エンドポイント `POST /api/cases/folders` を新設（既存 `api_add_case` にモード分岐を混ぜない）。
  - 入力: `parent_path` + `folder_name`。

## 課題④a PDFアイコン右クリックで別タブ表示

### 修正
- `static/doc.html` 新設: `?case=&sha=&page=` を読み、`viewer.js` の ViewerInstance をフルページでマウントするスタンドアロンビューア。
  - ctx スタブ: `onClose = window.close`、前後文書ナビは `/api/cases/{id}/documents` を取得して渡す。
  - **注釈は閲覧専用**にする（本体タブと同一ユーザーが同時編集すると D9 楽観ロックが自分対自分で409になり編集が失われるため。用途＝複数文書の見比べに合致）。
- `app.js`: 各行（kind=pdf）にPDFアイコン列を追加。実体は `<a href="/static/doc.html?case=…&sha=…" target="_blank" rel="noopener">`（⌘クリック・ミドルクリックも無料で対応）。加えて `contextmenu` で `preventDefault` → 直接 `window.open`（弁護革命の「右クリック→別タブ」の操作感を再現）。行クリックのインライン展開と衝突しないよう `stopPropagation`。
  - 【確認事項2】右クリック直開きはブラウザ標準メニュー（リンクをコピー等）を潰す。標準メニュー優先なら contextmenu ハンドラを外す（リンクなので標準メニューから「新しいタブで開く」可能）。デフォルトは依頼どおり直開きで実装。
- サーバ側追加なし（既存 GET API 再利用、GETのみのため新規セキュリティ面なし）。sha256 ベースURLは再索引後も有効（D2）＝期日メモに永続リンクを貼れる副産物。
- 進捗ログに「インライン展開仕様の置換ではなく追加であり D 逸脱に当たらない」旨を1行記録（遵守事項6）。

## 課題④b ドラッグ&ドロップでファイル追加【最後・要件充足までリリースしない】

原本フォルダへの書き込みを初めて許す機能。刑事弁護人レビューの必須3点（事件名明示確認・同名別内容の人間判断・インポートログ）が揃うまで出さない。

### サーバ: `POST /api/cases/{id}/upload`（multipart）
- 検証: CSRFヘッダ（既存ミドルウェアで自動適用）／拡張子ホワイトリスト＝`OPENABLE_EXTS`（小文字化後に照合）／ファイル名サニタイズ（basename化・パス区切り/NUL/`..`/先頭ドット除去・空文字化したら拒否）／解決後パスが事件フォルダ配下かつ **`_index/` 配下でない**ことを検証（`safe_resolve` 流用）／サイズ上限（500MB）。
- 書き込みは `.tmp` → `os.replace`（D9）。
- **重複・衝突の扱い（自動連番にしない）**:
  - アップロード内容の sha256 が既存文書と一致 → **追加拒否**し「既に登録済み（○○.pdf と同一内容）」と応答。
  - 同名・別内容 → 409 で「同名の別内容ファイルがあります（差し替え証拠の可能性）」を返し、フロントで人間に判断させる（別名で保存 or 中止。黙った連番付与はしない）。
- **インポートログ**: `_index/import_log.json` に追記式で `{時刻, ユーザー, ファイル名, sha256, サイズ}` を記録（検察官開示証拠と弁護人収集資料の出所区別・記録の同一性説明のため）。アトミック書き込み。
- 成功時に `_listing_cache.pop(case_id)`（これをしないと TTL 5秒の間、一覧に出ず再ドロップ→事故）。
- 保存先: 事件フォルダ直下を既定。【確認事項3】弁護革命エクスポートは `__Document__` 直下フラットのため、事件ごとに保存先サブフォルダを `cases.json` で持つか確認。

### フロント
- `docs-table` 上に dragover/drop ハンドラ＋ドロップゾーン表示。
- 確認ダイアログ: **最上部に「事件名：○○○○ に以下を追加します」を太字表示**（別事件への誤投入＝守秘義務事故の防止が最重要）、その下にファイル名一覧。「取り消す場合は Finder で削除→再索引」の注記（D13整合）。
- 応答に「Drive同期はバックグラウンドで行われます」の一行（同期遅延の注意喚起）。
- アップロード後 `loadDocuments()` → 未索引バッジ付きで即一覧表示（D5の既存経路に乗る。符号・作成者・日付の自動割り振りは課題②のパーサ修正で自動的に効く＝解析関数は1本に集約）。
- **未索引件数の可視化**: 検索0件時の注記に「未索引ファイル n 件は検索対象外」を併記（佐藤さん追加分を米谷側が再索引し忘れて『検索ヒットなし＝存在しない』と誤信する事故の防止）。

### テスト
- トラバーサル拒否・拡張子拒否・`_index` 書き込み拒否・CSRF必須・sha重複拒否・同名別内容409・インポートログ追記・キャッシュ無効化。
